For the battery: put `build-battery.mjs` in a `scripts/` folder and run `node scripts/build-battery.mjs`. It hits the World Bank's free, keyless API and writes two files in whatever directory you run it from:

- `battery.csv` — one row per (country, indicator) for 2022, with `id`, `truth`, `units`, `source`, `as_of`, and a `notes` column holding the plain-language question ("Total population — Sweden, 2022"). I verified the parsing, the null-skipping, the CSV quoting, and that the output loads cleanly through `loadSplit` with the meta attached.
- `estimates-template.csv` — the same ids with empty `observer_a/b/c` columns to fill.

The collection workflow, which is the actual rainy-day work: open `battery.csv`, and for each row ask each observer the question in its `notes` (rename the template columns to your real observers — `gpt5`, `claude`, `gemini`, a human panel, whatever). Put their numbers in `estimates-template.csv`, save, then load `battery.csv` + your filled estimates in **split mode**. That's the real version of the synthetic example — and if AI models really are ~0.97 correlated, you'll see the ensemble bar sit right next to the single bar, on your own data.

Three things worth knowing. The config block at the top (year, twelve indicators, twenty countries across scales) is meant to be edited — the cross-product is ~240 items, plenty to subset. The `YEAR = 2022` is deliberate: it predates common training cutoffs, which is the *controlled* condition (a model reciting a figure it memorized isn't estimating); build a second battery with a recent year for the *ecological* complement and read them side by side, exactly the split the framework's own roadmap names. And the CO2 indicator code is the one most likely to have moved — if it logs "0 values," check the current code in the World Bank catalogue and swap it in; everything else is long-stable.

The reason this is a builder and not a checked-in CSV is the same discipline the README preaches: the truth comes from the database with a date attached, not from my memory, so it's correct and reproducible. When you've run it and collected a couple of real observers, the obvious next build is the phase-2 observer-portfolio QP — "given how these specific observers actually correlate, which subset is worth consulting" — which is where your solver work re-enters.
